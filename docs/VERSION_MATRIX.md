The "label" is the user-facing version group; LivingWorld releases a new
label when the wire protocol changes for either edition. Patches within
one group share the same protocol number; their differences are
expressed through the capability bitset.

## How the matrix is used

- **Login negotiation** (`internal/version.Resolve`): when a client
  connects, the Java or Bedrock login handler calls
  `Resolve(edition, protocol)`. An exact match is required — unsupported
  protocols get a clean disconnect with the version matrix as the
  reference for "what's supported" (Master_Plan §6 Phase 2 / Master_Plan
  §3 audit row "First-party tests").
- **`/lwversion` in-game**: any player (OP or not) can run `/lwversion`
  to print the current label + protocol numbers + accepted builds. This
  is the player-facing diagnostic when a friend gets a "wrong version"
  disconnect.
- **`livingworld --version`**: same data, machine-friendly
  (tab-separated) for scripts and CI.
- **`livingworld-versioncheck`**: polls the upstream Mojang manifest
  against the local Java protocol number, exits non-zero when the
  matrix has drifted behind an official client.

## Adding a new letter (e.g. 26 (B))

When Mojang or Mojang Studios releases a new protocol number for the
supported year, bump the registry by adding a new `LWVersion` entry to
`internal/version/version.go`:

```go
RegisterVersion(LWVersion{
    Label:           "26 (B)",
    JavaProtocol:    776,
    JavaBuilds:      []string{"1.21.3"},
    BedrockProtocol: 976,
    BedrockBuilds:   []string{"1.26.30"},
    Capabilities:    CapCustomPlayerModels,
    ChangelogURL:    "https://github.com/Aofsnorth/LivingWorld/releases/tag/v26-B",
})
```

Clients on the previous letter are still served: `Resolve` returns the
matching entry; `Current()` returns the first registered (priority
order). `versioncheck -json` and the `--version` matrix both list every
entry, not just the current one.

## Capability bitset

| Bit name                  | Meaning                                                                                       |
|---------------------------|-----------------------------------------------------------------------------------------------|
| `CapCustomPlayerModels`   | The Java 1.21.2+ `custom_model_data` dimension split and Bedrock 1.21+ item component overlay are understood. Multiprotocol chains (Phase 7b) consult this. |
| `CapNetherUpdateNotes`    | The 1:8 nether coordinate scale is enforced client-side; server-side enforcement is unaffected. |

Adding a new bit is a public-API change: assign the next unused index
(bit 2, bit 3, …) and update this table in the same change set.
